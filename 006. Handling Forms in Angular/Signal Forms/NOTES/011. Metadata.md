If you want to attach additional information to form fields other than what the validation rules provide, then there is a powerful Metadata system that is offered in Angular's Signal Forms.

Common use cases include HTML input attributes (min, max, maxlength, pattern), custom UI hints (placeholder text, help text), and accessibility information.

There are six pre-defined metadata keys that are automatically populated by the validation rules -

Signal Forms provides six pre-defined metadata keys that validation rules automatically populate:

    REQUIRED - Whether the field is required (boolean)
    MIN - Minimum numeric value (number | undefined)
    MAX - Maximum numeric value (number | undefined)
    MIN_LENGTH - Minimum string/array length (number | undefined)
    MAX_LENGTH - Maximum string/array length (number | undefined)
    PATTERN - Regular expression pattern (RegExp[] - array to support multiple patterns)

When you use validation rules like 'required()' or' min('), they automatically set the corresponding metadata.

We can read these built-in metadata directly using the built-in accessors on the FieldState such as 'age().min()', 'username().required()', etc.

# CREATING CUSTOM METADATA KEYS

What if we want to create some custom metadata keys?

Let's say for our email field, we want to show the user some help text below the field such as -

    "We never share your email with anyone else."

Well, we can create a custom metadata key and then set its value when we create a new 'form'. First, we create a new key using the 'createMetadataKey' function -

    export const HELP_TEXT = createMetadataKey<string>();

Angular suggests that the metadata keys should be defined at module level and not inside the component class. Metadata keys rely on object identity, and recreating them loses that identity. That's why we define the 'HELP_TEXT' key outside the component class. The generic type parameter '<string>' indicates that the value of this metadata will be of type string.

Then we can set the value of this metadata key when we create a new form -

    loginForm = form(this.formModel, (schemaPath) => {
        metadata(schemaPath.email, HELP_TEXT, () => 'We never share your email with anyone else.');
    });

If you want to read this metadata, you can call the 'metadata' function and pass the key -

    const helpText = loginForm.email().metadata(HELP_TEXT);

Do note that the 'metadata' function returns a signal so you have to use it like -

    helpText() // to get the current value of the metadata

Let's say we want to display this help text in our template below the email field. We can simply do -

    @if (helpText) {
        <p class="text-gray-600 text-xs italic">{{ helpText() }}</p>
    }

This check is required because 'metadata' returns either the value of the metadata or 'undefined' if the metadata is not set. So we need to check if 'helpText' is defined before trying to display it in the template.

You might say, what if we set the value of a metadata key more than once?

    loginForm = form(this.formModel, (schemaPath) => {
        metadata(schemaPath.email, HELP_TEXT, () => 'We never share your email with anyone else 1.');
        metadata(schemaPath.email, HELP_TEXT, () => 'We never share your email with anyone else 2.');
        metadata(schemaPath.email, HELP_TEXT, () => 'We never share your email with anyone else 3.');
        metadata(schemaPath.email, HELP_TEXT, () => 'We never share your email with anyone else 4.');
    });

Well, by default, custom metadata keys use a "last write wins" strategy - if you call metadata() multiple times with the same key, only the last value is kept. So, in this case, the value of 'helpText' will be "We never share your email with anyone else 4.".

# METADATA REDUCER

Sometimes, instead of only keeping the last value, we might want to combine all the values in some way. For example, let's say we want to combine all the help text values into a single string separated by line breaks. In that case, we have to make sure we get all the values of the metadata key instead of just the last one.

There is a way to achieve this and for this, Angular provides a 'MetadataReducer' that we can pass to the 'createMetadataKey' function when we create a new metadata key. To understand how this works, we need to look at the type of the 'createMetadataKey' function. There are actually two ways in which we can call the 'createMetadataKey' function.

The first is what we already used -

    function createMetadataKey<TWrite>(): MetadataKey<Signal<TWrite | undefined>, TWrite, TWrite | undefined>;

Here, we do not pass anything to the 'createMetadataKey' function and all we provide it is the type argument 'TWrite' which is the type of the value that we want to store in this metadata key.

There is another way to call this function -

    declare function createMetadataKey<TWrite, TAcc>(reducer: MetadataReducer<TAcc, TWrite>): MetadataKey<Signal<TAcc>, TWrite, TAcc>;

This one takes a 'MetadataReducer' as an input and as we can also see, instead of only one type argument, it takes two type arguments - 'TWrite' which is the type of the value that we want to store in this metadata key and 'TAcc' which is the type of the accumulated value that is returned by the reducer.

This is how we generally use it -

    export const HELP_TEXT = createMetadataKey<string, string[]>(MetadataReducer.list());

Here, we are creating a metadata key that can store string values but instead of just keeping the last value, we want to keep a list of all the values. So we use the built-in 'MetadataReducer.list()' which simply accumulates all the values into an array.

Let's say now, we have the help_text on 'email' set multiple times like this -

    loginForm = form(this.formModel, (schemaPath) => {
        metadata(schemaPath.email, HELP_TEXT, () => 'Email should be unique.');
        metadata(schemaPath.email, HELP_TEXT, () => 'We never share your email with anyone else');
        metadata(schemaPath.email, HELP_TEXT, () => 'Email should be valid.');
    });

Now, since we are using a reducer, what gets returned from the 'metadata' function is not just the last value but the accumulated value which in this case is an array of all the values -

    const helpText = loginForm.email().metadata(HELP_TEXT);

So, 'helpText' is no longer a 'string' but rather a 'string[]' because of the reducer that we used. So if we want to display this in the template, we can do something like this -

    @if (helpText) {
        <ul>
            @for(text of helpText()) {
                <li class="text-gray-600 text-xs italic">{{ text }}</li>
            }
        </ul>
    }

And now, we will see all three help text values displayed in a list below the email field.

And we just looked at one built-in reducer that Angular provides - 'list'. This helps us accumulate all the values into an array. But this is not the only one. You also have -

    MetadataReducer.list() - Accumulates values into an array
    MetadataReducer.min() - Keeps the minimum value
    MetadataReducer.max() - Keeps the maximum value
    MetadataReducer.or() - Logical OR of boolean values
    MetadataReducer.and() - Logical AND of boolean values

# MANAGED METADATA KEYS

There is another concept under 'Metadata' in Angular's Signal Forms called 'Managed Metadata Keys'. What if you want to compute some new value from the accumulated result? In that case, instead of simply using 'createMetadataKey', you can use the 'createManagedMetadataKey' function.

There are two ways to call it. First is -

    function createManagedMetadataKey<TRead, TWrite>(create: (s: Signal<TWrite | undefined>) => TRead): MetadataKey<TRead, TWrite, TWrite | undefined>;

This means takes a 'create' function as an argument which accepts a signal of the accumulated value and returns the managed value based on it. Here, we have to pass two type arguments to this function -

    1. TRead - What type of value we want to read from this metadata key
    2. TWrite - What type of value we want to write to this metadata key

So, 'TWrite' is the type that should be compatible with what you pass to the 'metadata' function while setting the value of this metadata key. 'TRead' is the type that is returned when you call the 'metadata' function to read the value of this metadata key.

So, for example, I can say -

    export const HELP_TEXT  = createManagedMetadataKey<Signal<string[]>, string>((inputSignal) => {
        return computed(() => {
            const input = inputSignal() || '';
            return ['Extra help text', input]
        });
    });

Here, we are creating a managed metadata key where the type of value that we write is 'string' but the type of value that we read is 'Signal<string[]>'. The reason why we read a signal here is because the 'create' function that we pass to 'createManagedMetadataKey' receives a signal as an argument and we can use that signal to compute a new value which in this case is an array of strings. So, when we read this metadata key, we will get a signal of string array and whenever the value of this metadata key changes, the computed function will re-run and give us the updated array of strings.

Of course this is an example that does not make much sense but it is just to illustrate the concept of managed metadata keys. You can use this concept to create more complex metadata values that are computed based on the input value. Also, note that here, only the last value is kept because we are not using any reducer. So, if we set the value of this metadata key multiple times, only the last value will be used to compute the managed value.

Another way to call this function is like this -

    function createManagedMetadataKey<TRead, TWrite, TAcc>(create: (s: Signal<TAcc>) => TRead, reducer: MetadataReducer<TAcc, TWrite>): MetadataKey<TRead, TWrite, TAcc>;

In this case, there is a second argument to this function which is the reducer. This reducer determines how values accumulate. In the previous example, we did not use any reducer so by default, it only kept the last value. But if we want to keep all the values and compute a new value based on all the accumulated values, then we can use a reducer.

This is useful when you want to compute a value based on the accumulated value instead of just the last value. So, in this case, the 'create' function receives a signal of the accumulated value and we can use that to compute a new value. For example, maybe we want to return the count of the help text values alongside the accumulated list of help text. In that case, we can do something like this -

    export const HELP_TEXT  = createManagedMetadataKey<Signal<{ count: number; helpTexts: string[] }>, string, string[]>((inputSignal) => {
        return computed(() => {
        const helpTexts = inputSignal() || [];
        return {
            count: helpTexts?.length ?? 0,
            helpTexts
        }
        });
    }, MetadataReducer.list());

It seems like a lot is happening here but it is actually quite simple.

We are creating a managed metadata key where the type of value that we write is 'string', the type of value that we read is 'Signal<{ count: number; helpTexts: string[] }>' and the type of the accumulated value is 'string[]'.

We are using the 'MetadataReducer.list()' to accumulate all the values into an array and then in the 'create' function, we are computing a new value which is an object that contains the count of help texts and the list of help texts. Do note that the 'inputSignal' that we receive in the 'create' function is a signal of the accumulated value which in this case is 'string[]'.

So, whenever we set a new value for this metadata key, the computed function will re-run and give us the updated count and list of help texts.
