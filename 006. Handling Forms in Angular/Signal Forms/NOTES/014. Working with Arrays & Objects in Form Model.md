So far, we have seen a Form Model with a flat structure, meaning that all the form controls are at the same level -

    formModel = signal<FormData>({
        firstName: '',
        lastName: '',
        age: null,
        username: '',
        email: '',
        password: '',
        confirmPassword: '',
        stepCount: 0
    });

But, in a real-world application, we often need to work with more complex data structures, such as nested objects and arrays. Angular's Signal Forms provide robust support for handling these complex structures, allowing us to create dynamic and flexible forms.

# WORKING WITH NESTED OBJECTS

A very simple example of a nested object can be the 'name' field, which can be represented as an object containing 'firstName' and 'lastName'. Well, we can easily modify our form model to accommodate this structure. We can update the interface to reflect the nested structure and then initialize our form model accordingly -

    export interface FormData {
        name: {
            firstName: string;
            lastName: string;
        };
        age: number | null;
        username: string;
        email: string;
        password: string;
        confirmPassword: string;
        stepCount: number;
    }

    formModel = signal<FormData>({
        name: {
        firstName: '',
        lastName: ''
        },
        age: null,
        username: '',
        email: '',
        password: '',
        confirmPassword: '',
        stepCount: 0
    });

In the template, if we were previously referring to the 'firstName' and 'lastName' using -

    loginForm.firstName
    loginForm.lastName

We just have to update that to instead do -

    loginForm.name.firstName
    loginForm.name.lastName

And the same is applicable for the component class.

So, when to use a 'flat' structure and when to use a 'nested' structure?

Use flat structures when:

    1. Fields don't have clear conceptual groupings
    2. You want simpler field access (userForm.city vs userForm.address.city)
    3. Validation rules span multiple potential groups

Use nested structures when:

    1. Fields form a clear conceptual group (like an address)
    2. The grouped data matches your API structure
    3. You want to validate the group as a unit

# WORKING WITH ARRAYS

One of the most common use cases of Reactive Forms in Angular was 'Form Arrays' which allowed us to work with an array of form controls. This is especially useful when we have a dynamic number of form controls, such as a list of phone numbers or a list of addresses.

Let's say to our Signal Form, we want to add a functionality where a user can add one or more phone numbers. We will have a button to add a new phone number field, and each phone number field will have a corresponding remove button to delete that specific field.

First, we have to update the interface that is used by the Form Model -

    export interface FormData {
        name: {
            firstName: string;
            lastName: string;
        };
        age: number | null;
        username: string;
        email: string;
        password: string;
        confirmPassword: string;
        stepCount: number;
        phoneNumbers: string[];
    }

So, it now has a 'phoneNumbers' field which is an array of strings. Next, we have to initialize this field in the Form Model -

    formModel = signal<FormData>({
        name: {
        firstName: '',
        lastName: ''
        },
        age: null,
        username: '',
        email: '',
        password: '',
        confirmPassword: '',
        stepCount: 0,
        phoneNumbers: []
    });

Initially, let's say we will have no phone number field. So, we initialized the array as empty.

Now, in the HTML, we have to iterate over this array and then for each element in the array, we will have a separate phone number input field, just like how we used to do in Reactive Forms -

    @for(phoneNumber of loginForm.phoneNumbers; let index = $index; track index) {
        // The input field, error message and other blocks for each phone number input field
    }

And now, let's also add some buttons, one in front of each phoneNumber input that is used to delete the field. And one will be there to add new inputs for phone number.

    <button class="bg-blue-500 hover:bg-blue-700 text-white font-bold py-2 px-4 rounded focus:outline-none focus:shadow-outline cursor-pointer" type="button" (click)="addPhoneNumber()">
        Add phone number
    </button>

So, to add a phone number, we just have to update the formModel because it is a writable signal -

    addPhoneNumber() {
        this.loginForm.phoneNumbers().value.update(phoneNumbers => [...phoneNumbers, '']);
    }

Well, we simply get the current value of the phoneNumbers array, spread it to create a new array and then add an empty string at the end which will represent the new phone number field. We can do this because remember that 'value' gives us a Signal and 'signal' has an 'update' method that allows us to update the value of the signal based on its current value.

Similarly, let's say we have a button to delete an input field for phone number -

    <button (click)="removePhoneNumber(index)">x</button>

So, just like adding, we can remove a field like this -

    removePhoneNumber(index: number) {
        this.loginForm.phoneNumbers().value.update(phoneNumbers => phoneNumbers.filter((number, i) => i !== index));
    }

Again, we used the 'update' method to update the array of phone numbers. Remember that if we directly update the formModel signal by doing -

    this.formModel().phoneNumbers = [...this.formModel().phoneNumbers, ''];

This won't reflect the change in the form because here, we are not updating the signal, we are just updating one of the properties of the object that is held by the signal.

Yes, we could've used the formModel to update the phone numbers array like this -

    this.formModel.update(formModel => {
      return {
        ...formModel,
        phoneNumbers: [...formModel.phoneNumbers, '']
      }
    });

But this is more code when compared to just doing -

    this.loginForm.phoneNumbers().value.update(phoneNumbers => phoneNumbers.filter((number, i) => i !== index));

# VALIDATIONS ON ARRAYS

Alright, finally, let's understand how we can add some validation logic to all the fields in an array. Let's say that for all the phone numbers, we want to ensure they are 'required' (if a user adds a field), and they follow a phone number pattern, meaning we accept only valid phone numbers as inputs.

Normally, if we had only one phone number field, we would do -

    required(schemaPath.phoneNumber, { message: 'Phone number is required' });
    pattern(schemaPath.phoneNumber, /[0-9]{10}/, { message: 'Phone number is not valid!'});

But, here, we have an array of phone number fields. So, for each field, we want these two validation rules to apply. Well, we can do that using the 'applyEach' method -

    applyEach(schemaPath.phoneNumbers, (phoneNumber) => {
      required(phoneNumber, { message: 'Phone number is required' });
      pattern(phoneNumber, /[0-9]{10}/, { message: 'Phone number is not valid!'});
    });

The 'applyEach' method is of two types -

    function applyEach<TValue extends ReadonlyArray<any>>(path: SchemaPath<TValue>, schema: NoInfer<SchemaOrSchemaFn<TValue[number], PathKind.Item>>): void;

    function applyEach<TValue extends Object>(path: SchemaPath<TValue>, schema: NoInfer<SchemaOrSchemaFn<ItemType<TValue>, PathKind.Child>>): void;

So, either it takes an array or an object. In our case, since we are working with an array of phone numbers, we will be using the first one. The 'applyEach' method will apply the validation rules defined in the second argument to each item in the array specified by the path in the first argument.

And that's it! This is how we work with nested objects and arrays in Angular's Signal Forms. This allows us to create dynamic and flexible forms that can handle complex data structures with ease.
