In the last section, we learned how to use some of the built-in rules/methods to control the field state such as 'disabled', 'hidden' and 'readonly'.

Now, there is one very useful built-in method that we haven't discussed yet and that is 'debounce'.

First, let's understand what even is debouncing. Debouncing is a technique used to limit the rate at which a function gets called. It ensures that a function is not called too frequently, especially in response to events that can occur rapidly, such as user input or window resizing.

An example of this in our case is the 'username' field. Let's say we have a validation rule that checks if the username is already taken or not. By default, this validation will run each time the user types a character. It means, if the username is 8 characters long, the validation will run 8 times. This can lead to performance issues and unnecessary server calls if we are making an API request to check the username availability. Ideally, we should only check for the username availability when the user has stopped typing for a certain amount of time, let's say 500 milliseconds. This is where debouncing comes into play.

In Signal Forms, we have a 'debounce' rule that we can use to achieve this. The simplest way to use it is to pass it the SchemaPath and the debounce time in milliseconds. This is the type of the 'debounce' function -

    function debounce<TValue, TPathKind extends PathKind = PathKind.Root>(path: SchemaPath<TValue, SchemaPathRules.Supported, TPathKind>, config: number | 'blur' | Debouncer<TValue, TPathKind>): void;

So, it takes a 'SchemaPath' as the first argument like other built-in rules, and the second argument is a 'config' which is either a number representing the debounce time in milliseconds, the string 'blur' which means delay model synchronization until the field loses focus, or a custom 'Debouncer' function for more complex debouncing logic.

For example, if we want to debounce the 'username' field for 500 milliseconds, we can do it like this -

    debounce(schemaPath.username, 500);

With a 500ms debounce:

    1. User types in the input field
    2. Form model updates only after 500ms of typing inactivity
    3. If user keeps typing, the timer resets with each keystroke
    4. Once user pauses for 500ms, the model updates with the final value

You might argue that what happens if the timer has not yet completed and the user submits the form? Or maybe the user is done typing and then maybe clicks outside? Well, Angular forms handle such scenarios beautifully.

The debounce() function ensures users don't lose data through these mechanisms:

    1. When marked as touched: The value syncs immediately, aborting any pending debounce delay. This happens when the field loses focus (blur) or when explicitly marked as touched.

    2. On form submission: All fields are marked as touched before validation, which ensures all debounced values sync immediately.

This means users can type quickly, tab away, or submit the form without waiting for debounce delays to expire.

# 'blur' OPTION

Instead of a number for the debounce time, we can also pass the string 'blur' as the second argument. This means that the model will only be updated when the field loses focus. For example, if we write -

    debounce(schemaPath.username, 'blur');

This means, the 'username' field will only update the form model when the user clicks outside the input field or tabs away from it. This can be useful in scenarios where we want to avoid unnecessary updates while the user is still typing, but we want to ensure that the model is updated as soon as the user is done with the input.

The benefit of this over a time-based debounce is that it provides a more natural user experience. Users often expect that their input will be processed once they move on to the next field or click away, rather than having to wait for a timer to expire.

# CUSTOM DEBOUNCE LOGIC

Sometimes, we want more control over the debouncing behavior than just specifying a duration. In such cases, the second argument can be passed as a function instead of a simple duration. This function is called a 'Debouncer' and it allows us to implement custom debouncing logic.

This is the type of this 'Debouncer' -

    type Debouncer<TValue, TPathKind extends PathKind = PathKind.Root> = (context: FieldContext<TValue, TPathKind>, abortSignal: AbortSignal) => Promise<void> | void;

So, it is a function that takes a 'FieldContext' which is basically the same context object that we get in the validation functions, and an 'AbortSignal' which can be used to cancel the debouncing if needed. The function can return a 'Promise' or void.

    debounce(username, () => {
      return new Promise((resolve) => {
          setTimeout(() => resolve(), 500);
      });
    });

Of course here we are doing the same thing as passing a duration of 500ms, but this way we can implement more complex debouncing logic if needed.

As per Angular's documentation, some of the common use-cases of the custom 'Debouncer' function can be -

    1. Implementing custom timing logic beyond simple delays
    2. Coordinating synchronization with external events
    3. Conditional debouncing based on application state

Debouncing is most useful when:

    1. You have expensive computed signals that depend on the field value
    2. The field triggers API calls or other side effects
    3. You want to reduce validation overhead during rapid typing
    4. Performance profiling shows model updates are causing slowdowns

Don't use debouncing if:

    1. The field needs immediate updates for good UX (such as calculator inputs)
    2. The performance benefit is negligible
    3. Users expect real-time feedback
