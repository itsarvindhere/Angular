So far, we have discussed synchronous validation in Signal Forms. This means, validations that are performed without waiting for something e.g. an api call's response.

There are times when we have to perform asynchronous validations. For example, a user might enter some username but we have to check if that username is available or not. It is not possible to check that synchronously so we have to make an API call for that.

How do we handle such validations in signal form?

There is a built-in function provided by Angular for signal forms that we can use to perform http validations. This function is named 'validateHttp()' -

    function validateHttp<TValue, TResult = unknown, TPathKind extends PathKind = PathKind.Root>(path: SchemaPath<TValue, SchemaPathRules.Supported, TPathKind>, opts: HttpValidatorOptions<TValue, TResult, TPathKind>): void;

So, this function also takes the 'SchemaPath' as the first argument but the main difference here is the second argument which is of type 'HttpValidatorOptions'. This is an interface with these properties -

    1. readonly request: ((ctx: FieldContext<TValue, TPathKind>) => string | undefined) | ((ctx: FieldContext<TValue, TPathKind>) => HttpResourceRequest | undefined);
    2. readonly onSuccess: MapToErrorsFn<TValue, TResult, TPathKind>;
    3. readonly onError: (error: unknown, ctx: FieldContext<TValue, TPathKind>) => TreeValidationResult;
    4. readonly options?: HttpResourceOptions<TResult, unknown>;

So, in the options object, we pass what is the 'url' to which we want to make the request, what happens when the api call is successful, what happens when it fails, and other options.

Let's take an example of a validator which makes an api call to check if a username is available. Let's say we make a call to - 

    /api/check-username?username=<the username we want to check>

So, this is how our validator will look like -

    checkUsername(schemaPath: SchemaPath<string>, options?: {message?: string}) {
        validateHttp(schemaPath, {
            request: ({value}) => `api/check-username?username=${value}`,
            onSuccess: (response: any) => {
                if (response.taken) {
                    return {
                        kind: 'usernameTaken',
                        message: 'Username is already taken.'
                    };
                };
                return null;
            },
            onError: (error) => {
                return {
                    kind: 'networkError',
                    message: 'Could not verify username availability',
                };
            }
        });
    };

So, here, we created an asynchronous validator which makes an api call to 'api/check-username?username=${value}' and here, the 'value' is coming from the 'FieldContext' type argument which the function passed to 'request' property accepts.

If the API call is successful, we check its 'response' and if 'response' has a 'taken' property set to 'true', it means the username is already taken so we return an error object. Otherwise, it means that the username is available so we return no error object and instead return 'null'.

If the API call results in an error, then also we return an error object telling the user about the network error.

Do note that Asynchronous validation runs only when synchronous validation passes successfully. So, if we have one or more synchronous validators on the username field, then our Asynchronous Validation will not run unless all the synchronous validators pass.

Another thing about Asynchronous Validators is that since they take some time to run and return some result, we can show some message to the user while the validators are running using the 'pending()' signal on the form state.

    @if (form.username().pending()) {
        span class="spinner">Checking...</span>
    }
